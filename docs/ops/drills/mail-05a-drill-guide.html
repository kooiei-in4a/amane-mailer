Shared Mailer Deployment Drill Guide

Deploy host 上の shared Mailer で、送信なし smoke と ACS 1 通ドリルを安全に再実行するための手順です。 production ACS 常時有効化前の no-send / ACS drill 用で、通常状態の ACS_CONNECTION_STRING が空であることを前提にします。 connection string、Bearer token、実宛先は GitHub issue、PR、ログ、コミットに残さないでください。

/opt/amane-mailer develop tenant only staging / production fail-closed
Safety gate: ACS ドリルでは一時 tenant JSON で develop だけ live_sending=true にし、 staging と production は明示的に live_sending=false にします。 スクリプトは復旧 trap を持ち、通常 compose へ戻してから終了します。 production ACS がすでに有効な環境では、この手順を汎用ドリルとして使わず、別のレビュー済み runbook を作成してください。
Current status: helper scripts use the SQLite Mailer CLI for DB verification and a temporary curlimages/curl compose service for HTTP calls. They do not depend on a PostgreSQL db service, psql, or shell tools inside the chiseled Mailer runtime image.

Before You Run

  • 作業ディレクトリは /opt/amane-mailer。
  • ACS_CONNECTION_STRING は通常状態で空。
  • develop アプリ側の送信フローがドリル中に動かないことを確認済み。
  • mail_requests に queued / processing が残っていない。
  • production tenant の queued / processing が 0。
  • 検証用 recipient は 1 件だけ。
  • bash -x や secret 入力周辺の shell tracing は使わない。
  • この手順は develop tenant 固定。staging / production は別 script またはレビュー済み override を使う。

No-Send Smoke

shared Mailer の health / readiness、401 auth smoke、202 API 受付、DB 記録を確認します。 worker disabled 中に DB を確認するため、実送信も fail-closed 処理も進みません。 DB 確認は ./Amane.Mailer db request-state を使います。 

cd /opt/amane-mailer
bash mail-05a-no-send-smoke.sh

Expected Result

  • /healthz と /readyz が 200。
  • 不正 token は 401 UNAUTHORIZED_TENANT。
  • 正しい develop token は 202 accepted。
  • 401 request は found=false。
  • worker disabled 中の 202 request は status=queued、attempt_count=0、attempt_rows=0、provider_message_id_present=false。
  • restore 後、202 smoke request は通常 worker により fail-closed の非配送終端状態へ進む。これは想定どおり。
  • 終了時に compose.worker-disabled.yml、compose.mail-05a-client.yml、mail-05a-work/ が削除される。

ACS Live-Send Drill

ACS で実際に 1 通だけ送ります。スクリプトは一時 tenant JSON と compose override を作り、 --force-recreate で Mailer に drill 設定を読み直させます。 DB 確認は ./Amane.Mailer db stats と db request-state を使います。

Prepare Secret Input On Deploy Host

cd /opt/amane-mailer
cp mail-05a-acs-drill.input.example.json mail-05a-acs-drill.input.json
chmod 600 mail-05a-acs-drill.input.json
vi mail-05a-acs-drill.input.json

connection_string は endpoint=https://...;accesskey=... の全文です。 sender_email は ACS 検証済み sender、recipient_email は承認済み検証用宛先 1 件にします。

Run

bash mail-05a-acs-drill.sh

Expected Result

  • HTTP_STATUS=202 と accepted。
  • DB_STATE=status=delivered,attempt_count=1,attempt_rows=1,provider_message_id_present=true,...。
  • ACS_DELIVERED_BY_PROVIDER が出る。
  • 終了時に NORMAL_COMPOSE_RESTORED。
  • provider delivery 成功後、実値入り mail-05a-acs-drill.input.json は既定で削除される。
  • precheck failure や未完了 drill では、修正して再実行できるよう input file は残る。

Record

drill record、PR comment、または運用メモには、秘密情報なしで次だけ記録します。

  • 実施日時と MAILER_IMAGE_TAG。
  • request id。
  • API result: 202 accepted。
  • DB result: status、attempt count、attempt rows、provider message id present/absent。
  • 受信箱で 1 通確認できたこと。
  • 一時ファイル削除、通常 compose 復旧、ACS_CONNECTION_STRING 空、tenant path 通常値。

Troubleshooting

Mailer restarts during drill

docker compose logs --tail 120 mailer を確認します。 tenant JSON が読めない場合は bind mount 元ファイルがコンテナから読める権限か見ます。

LIVE_SENDING_DISABLED

tenant JSON 変更後に Mailer が再作成されていない可能性があります。 drill script は --force-recreate を使います。

ACS request failed

connection string、sender domain、ACS Email resource association、 recipient allow policy を Azure 側で確認します。秘密値はログに貼らないでください。

Timeout logs

ready timeout 時の recent Mailer logs はローカル調査用です。 GitHub issue / PR に貼る場合は endpoint、sender、recipient、token、provider details を先に伏せてください。

ACS precheck is not empty

ACS precheck が connection string の事前設定を検出した場合、この no-send / ACS drill の前提外です。 production ACS 有効後は別の runbook で扱います。

Recovery check

復旧後は /readyz、ACS_CONNECTION_STRING 空、 MAILER_TENANTS_PATH=/app/config/mailer/tenants.json を確認します。
